# Token-saving стратегия ANT

Цель `ant-compact` и `ant-low-words` — не просто писать меньше. Цель — сохранять действие, смысл и безопасность при меньшем количестве слов.

## Что реально экономим

1. **Output tokens** — слова, которые агент пишет пользователю.
2. **Conversation history** — короткие ответы меньше загрязняют будущий контекст.
3. **Human time** — пользователь быстрее понимает следующий шаг.

## Что не решается одной краткостью

Краткий стиль не всегда сильно снижает общую стоимость reasoning-моделей, потому что часть токенов может уходить на скрытое рассуждение, входной контекст, файлы, логи и tool outputs. Поэтому ANT Low Words полезен, но его не стоит продавать как магическое “-80% cost” без evals.

## Правильная формула

```text
fewer words + same next action + same safety = good compression
fewer words - safety = bad compression
fewer words - clarity = bad compression
```

## Что сжимать первым

1. Приветствия и похвалу.
2. Повтор запроса пользователя.
3. Общие фразы.
4. Длинную теорию.
5. Несколько вариантов, если есть один лучший.
6. “Technical appendix”, если не просили.

## Что не сжимать

1. Команды.
2. Где запускать команду.
3. Как понять, что сработало.
4. Предупреждения о риске.
5. Неопределённость, которая влияет на решение.
6. Названия файлов, переменных, API, ошибок.

## Будущая сильная фича

Добавить `ant-session-summary`:

- после длинной сессии агент пишет компактное резюме;
- сохраняет решения, файлы, команды, открытые вопросы;
- убирает болтовню;
- делает следующий запуск агента дешевле и понятнее.

Это может быть полезнее для стоимости, чем просто короткие ответы.
