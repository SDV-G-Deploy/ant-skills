# Режимы ANT

ANT лучше развивать не как один стиль, а как маленькую линейку режимов.

## 1. `ant-plain-language`

Основной режим.

Для кого: non-technical builders, начинающие разработчики, фаундеры, дизайнеры, продакты.

Цель: объяснить техническую работу человеческим языком, без потери точности.

Использовать, когда:

- человек не понимает ошибку;
- нужно объяснить команду, деплой, API, базу данных, auth, Git;
- нужно показать риски и проверку;
- пользователь учится строить проекты через AI.

## 2. `ant-compact`

Короткий рабочий режим.

Для кого: люди, которые уже немного понимают контекст и не хотят длинных объяснений.

Цель: сохранить понятность, но сократить слова.

Формула:

```text
Смысл → Действие → Проверка
```

Пример:

```text
Смысл: проект не нашёл библиотеки.
Действие: в папке проекта запусти `npm install`.
Проверка: команда закончилась без красных ошибок, потом `npm run dev` запускает сайт.
```

## 3. `ant-low-words`

Жёсткая экономия слов.

Для кого: опытные пользователи AI-агентов, длинные coding-сессии, частые статусы, дорогие токены.

Цель: максимум смысла, минимум слов.

Формула:

```text
Причина: ...
Фикс: ...
Проверка: ...
```

Пример:

```text
Причина: нет `.env`.
Фикс: создай `.env`, добавь `DATABASE_URL`.
Проверка: `npm run dev` без ошибки env.
```

## Почему не называть основной режим extreme shorthand

Некоторые пользователи просят "caveman style", когда имеют в виду экстремальную краткость, но для ANT это плохой главный бренд:

- может звучать грубо или насмешливо;
- плохо сочетается с идеей уважительного объяснения;
- хуже подходит для non-technical аудитории;
- может провоцировать агента писать слишком примитивно.

Лучше:

- публично: **ANT Compact** / **ANT Low Words**;
- внутренняя обработка: считать "caveman style" legacy-триггером для уважительной экстремальной краткости.

## Рекомендованная матрица

| Ситуация | Режим |
|---|---|
| Пользователь не понимает ошибку | `ant-plain-language` |
| Нужно коротко сказать, что делать | `ant-compact` |
| Длинная сессия, нужен минимум токенов | `ant-low-words` |
| Риск удалить/сломать/опубликовать | любой режим, но safety override |
| Пользователь просит урок | `ant-plain-language` |
| Пользователь просит “без воды” | `ant-compact` |
| Пользователь просит “максимально коротко” | `ant-low-words` |

## Важное ограничение

Короткие ответы экономят output tokens. Но они не решают всю проблему стоимости и контекста. В длинных coding-сессиях часто больше всего весят входной контекст, логи, файлы, история чата и tool outputs. Поэтому ANT должен развиваться в две стороны:

1. короткие ответы;
2. компактные статусы, summaries и memory-файлы.
